# Release and Compatibility Policy

This document defines a simple compatibility and release policy for the project.

## Current Compatibility Targets

The project currently targets:

- Python `3.12`
- Azure Functions runtime `v4`
- Azure Table Storage for operational configuration and state
- Azure Resource Graph for discovery

Current real action support:

- `Microsoft.Compute/virtualMachines`

## Versioning Direction

The project baseline is now `1.0.0`.

From `1.0.0` onward:

- breaking changes should be exceptional and clearly announced
- schema or behavior changes must include migration guidance
- examples and operational docs should be updated in the same change
- app settings, table schemas, and retain semantics should not change silently

Maintainer expectation:

- document breaking changes clearly
- update examples and developer docs together with the change
- avoid silent schema or behavior changes

## Table Schema Compatibility

The scheduler depends on 3 tables:

- `OffHoursSchedulerConfig`
- `OffHoursSchedulerSchedules`
- `OffHoursSchedulerState`

If a change affects entity shape, required fields, or interpretation rules:

- the change must be documented
- examples must be updated
- release notes or PR notes should call out migration impact

Important audit fields expected today:

- `Version`
- `UpdatedAtUtc`
- `UpdatedBy`

## Deploy Compatibility

The recommended deploy path is:

```bash
./scripts/deploy_scheduler.sh \
  --parameters-file infra/bicep/main.parameters.json
```

This is especially important when using:

- `managementGroupIds`
- `excludeSubscriptionIds`

Because the wrapper resolves the effective subscription scope before deployment.

## Dependency Policy

Current rule:

- root `requirements.txt` is the source of truth for Python runtime dependencies
- `function/requirements.txt` is generated by `scripts/prepare_function_app_publish.sh`

Contributors should update runtime dependencies in the root file and validate:

- tests
- function publish bundle preparation

## CI Expectations

Changes should not break:

- `pytest -q`
- shell script syntax validation
- Bicep compilation
- function bundle preparation

The repository CI is expected to validate these baseline checks.

## Backward Compatibility Guidance

The project tries to preserve:

- documented app settings
- documented table names
- the operational meaning of `DRY_RUN`
- the current retain semantics:
  - `RETAIN_RUNNING` as a temporary manual override
  - `RETAIN_STOPPED` as a sticky manual override

If a change must break compatibility:

- document it clearly
- provide migration guidance where possible

## Release Notes Guidance

When publishing a meaningful release or milestone, include:

- new capabilities
- breaking changes
- migration steps
- deploy changes
- schema changes
- compatibility notes
